# 뇌피셜 (Brain-fficial) — 개발자 가이드

> 최종 업데이트: 2026-03-09
> 기획서 v6 + 전략 논의 결과 통합

---

## 1. 프로젝트 개요

**뇌피셜**은 찰리 멍거의 25가지 인지 편향을 기반으로 한 AI 대화형 웹앱입니다.

- **Step 1:** AI 적응형 대화로 사용자의 인지 편향 Top 3 진단 (7~12턴)
- **Step 2:** 사용자의 실제 고민을 6개 렌즈(심리학, 경제학 등)로 찬반 분석

**핵심 철학:** "AI는 답을 주지 않는다. 더 좋은 질문을 준다."

## 3. 아키텍처

```
[브라우저]
  ├── 서버 컴포넌트: 랜딩, 결과 페이지 (SEO)
  └── 클라이언트 컴포넌트: 대화 UI, 대시보드 (인터랙션)
       │
       ▼
[Next.js API Routes]
  ├── POST /api/step1/chat      ← Gemini Flash-Lite
  ├── POST /api/step1/result    ← Gemini Flash-Lite
  └── POST /api/step2/analyze   ← Gemini Flash
       │
       ▼
[Firebase Firestore]
  ├── users/{uid}
  ├── users/{uid}/diagnoses/{id}
  └── users/{uid}/analyses/{id}
```

### 핵심 원칙

- **서버/클라이언트 분리:** SEO가 필요한 페이지(랜딩, 결과)는 서버 컴포넌트, 인터랙션이 필요한 페이지(채팅, 대시보드)는 클라이언트 컴포넌트
- **점수 계산은 서버사이드:** AI는 질문+선택지+`bias_map`만 반환. 점수 누적과 확신도 계산은 `src/lib/bias/scoring.ts`에서 처리 (AI 응답 일관성 확보)
- **진단 상태의 소유권은 서버:** Step 1/2의 원본 상태(`bias_scores`, 직전 선택지의 `bias_map`, 최종 Top 3, 분석 이력)는 서버 세션 또는 Firestore 문서가 기준이며, 클라이언트는 렌더링용 스냅샷만 보관
- **비로그인 데이터:** localStorage는 UX 복구용 캐시로만 사용. 정답 데이터는 서버의 익명 세션 문서에 저장하고, 로그인 시 Firestore 사용자 문서로 마이그레이션
- **익명 사용자 제한은 best-effort:** 비로그인 1회 제한은 브라우저 저장소 + 익명 세션 기준의 완화된 제한으로 처리. 강한 quota 보장은 로그인 사용자만 대상

## 5. API 명세

### 5-1. `POST /api/step1/chat` — 적응형 대화

**요청:**
```typescript
{
  session_id: string;           // 서버가 발급한 Step 1 세션 ID
  turn_number: number;          // 현재 턴 (1~12)
  selected_option_id?: string;  // 직전 턴에서 사용자가 고른 옵션 ID
  input_text?: string;          // 직전 턴의 짧은 자유입력 값
}
```

**응답:**
```typescript
{
  session_id: string;
  question: string;             // 질문 텍스트
  should_end: boolean;          // 서버 계산 결과
  progress: number;             // 0~100
  options: [
    {
      id: string;               // 서버가 추적하는 옵션 ID
      text: string;             // 선택지 텍스트
      type: "touch" | "input";  // 터치 선택 or 짧은 타이핑
    }
  ];
}
```

> **주의:** 클라이언트는 `bias_scores`, `top3_confidence`, `uncertain_biases`, `conversation_history`를 신뢰 가능한 입력으로 보내지 않습니다. 이 값들은 전부 서버가 세션 문서에서 읽어옵니다. AI에게는 내부적으로 `bias_map` 생성을 요청하되, 그 결과는 서버 세션에만 저장합니다.

**서버 후처리 (`scoring.ts`):**
```typescript
// 서버가 세션에 저장된 직전 options[].bias_map을 기준으로:
// 1. selected_option_id 검증
// 2. 기존 bias_scores에 누적
// 3. Top 3 확신도 재계산
// 4. should_end 판단 (Top 3 확신도 전부 0.80+ 또는 12턴)
// 5. progress 계산 (프론트 진행 바용)
// 6. 다음 질문/options 생성 후 options[].bias_map은 서버 세션에만 저장
```

### 5-2. `POST /api/step1/result` — 결과 생성

**요청:**
```typescript
{
  session_id?: string;           // 비로그인 Step 1 세션 ID
  diagnosis_id?: string;         // 로그인 사용자의 기존 진단 ID
}
```

**응답:**
```typescript
{
  top3: [
    {
      rank: number;
      bias_id: number;
      character_name: string;    // "끈기의 투자자"
      subtitle: string;          // "쉽게 포기하지 않는 사람"
      summary: string;           // 한 줄 요약
      detail: string;            // 2~3문단 (대화 사례 반영 필수)
      trigger_situations: string[];  // 발동 상황 4개
      famous_person: string;     // 대표 인물
      munger_quote: string;      // 관련 멍거 명언
    }
  ];
  radar_scores: Record<string, number>;
  overall_insight: string;       // 종합 인사이트 2~3문장
}
```

> **주의:** `final_scores`, `top3_biases`, `conversation_log`는 클라이언트 입력으로 받지 않습니다. 결과 생성 시 서버가 세션/진단 원본을 조회합니다.

### 5-3. `POST /api/step2/analyze` — 고민 분석

**요청:**
```typescript
{
  user_input: string;            // 고민 텍스트 (최대 5000자)
  diagnosis_id?: string;         // 로그인 사용자의 진단 ID
  session_id?: string;           // 비로그인 Step 1 세션 ID
}
```

**응답:**
```typescript
{
  selected_lenses: [
    {
      lens: string;              // "심리학"
      emoji: string;             // "🧠"
      verdict: "찬성" | "반대" | "조건부";
      reasoning: string;
    }
  ];
  excluded_lenses: [
    { lens: string; emoji: string; reason: string }
  ];
  bias_warning: {
    primary_bias: string;
    how_it_works: string;
    affected_lens: string;
    direction: string;
  };
  critical_questions: string[];  // 결정적 질문 2~3개
  closing: string;
}
```

> **주의:** `bias_profile`은 클라이언트가 직접 보내지 않습니다. Step 2 분석 시 서버가 Step 1 원본 결과에서 Top 3 편향 프로필을 구성합니다.

## 6. 데이터 모델 (Firestore)

```
users/{uid}
  ├── displayName: string
  ├── email: string
  ├── biasScores: number[]       // 25개 편향 점수 배열
  ├── top3: number[]             // Top 3 편향 ID
  ├── lastDiagnosisAt: Timestamp
  └── createdAt: Timestamp

users/{uid}/diagnoses/{id}
  ├── conversationLog: Message[]
  ├── finalScores: Record<string, number>
  ├── top3Result: BiasResult[]
  ├── radarScores: Record<string, number>
  └── createdAt: Timestamp

users/{uid}/analyses/{id}
  ├── userInput: string
  ├── selectedLenses: LensResult[]
  ├── biasWarning: BiasWarning
  ├── criticalQuestions: string[]
  └── createdAt: Timestamp

anonymousSessions/{sessionId}
  ├── type: "step1" | "step2"
  ├── status: "in_progress" | "completed"
  ├── biasScores: number[]
  ├── top3: number[]
  ├── top3Confidence: number[]
  ├── conversationLog: Message[]
  ├── currentOptions: StepOptionInternal[]   // bias_map 포함, 서버 전용
  ├── resultSnapshot?: Step1Result
  ├── step2Input?: string
  ├── step2Result?: Step2Result
  ├── quotaDate: string                      // YYYY-MM-DD
  └── expiresAt: Timestamp
```

### Security Rules 핵심
```
match /users/{uid}/{document=**} {
  allow read, write: if request.auth != null && request.auth.uid == uid;
}

match /anonymousSessions/{sessionId} {
  allow read, write: if false; // 서버(Admin SDK)만 접근
}
```

## 8. 핵심 구현 주의사항

### 8-1. AI 응답 안정성
- Gemini JSON 출력이 깨질 수 있음 → **JSON 파싱 실패 시 재시도 (3회, 지수 백오프)**
- `response_mime_type: "application/json"` 설정으로 JSON 모드 강제
- 그래도 실패하면 사용자에게 "다시 시도해주세요" 안내

### 8-2. 번들 최적화
```typescript
// Chart.js, html2canvas는 반드시 dynamic import
const RadarChart = dynamic(() => import('@/components/charts/RadarChart'), {
  ssr: false,
  loading: () => <ChartSkeleton />
});
```

### 8-3. 연타 방지
- **클라이언트:** 선택지 클릭 시 즉시 `disabled` + 디바운싱 (300ms)
- **서버:** 동일 세션의 중복 요청 차단 (`session_id + turn_number` idempotency key)

### 8-4. 비로그인 데이터 흐름
```
사용자 진입 → 서버가 anonymous session 발급
    → 진행 중: 서버 세션 문서 저장 + localStorage에는 session_id/임시 스냅샷만 저장
    → 결과 페이지: session_id 기준으로 서버에서 결과 조회
    → Step 2 결과도 anonymous session에 저장
    → 로그인 시: anonymous session의 Step 1/Step 2 데이터를 Firestore로 마이그레이션 → localStorage 캐시 삭제
```

### 8-5. 이탈 방지
```typescript
// Step 1 대화 중 페이지 이탈/새로고침 시 경고
window.addEventListener('beforeunload', (e) => {
  if (isInProgress) {
    e.preventDefault();
    e.returnValue = '';
  }
});
```

### 8-6. 입력 검증 (Step 2)
- 길이 제한: 5,000자
- HTML 태그 제거
- 프롬프트 인젝션 방어: 시스템 프롬프트와 사용자 입력 분리

### 8-7. 사용 횟수 제한
| 기능 | 비로그인 | 로그인 |
|------|----------|--------|
| Step 1 진단 | 1회/일 (best-effort) | 1회/일 |
| Step 2 분석 | 1회 (best-effort) | 3회/일 |
| 히스토리 | 없음 | 최근 10개 |

> **정책 메모:** 비로그인 제한은 localStorage 삭제, 시크릿 모드, 기기 변경으로 우회 가능. 제품 문구도 "체험은 제한될 수 있음" 수준으로 맞추고, 강한 제한은 로그인 사용자에게만 보장합니다.

## 11. 유지보수 원칙

이 프로젝트는 일반적인 CRUD 웹앱보다 **프롬프트, 세션 상태, 응답 스키마** 유지보수가 더 중요합니다. UI 변경보다도 AI 응답 형식, 점수 계산 로직, Firestore 문서 구조가 깨질 때 서비스 전체 신뢰도가 빠르게 떨어지므로 아래 원칙을 기본 운영 규칙으로 둡니다.

### 11-1. 단일 진실 원본 분리
- 이 문서는 제품/기술 개요를 설명하는 **상위 가이드**로 유지
- 실제 구현 기준은 아래 문서/코드가 단일 진실 원본(Single Source of Truth)
- API 계약: `src/types/*.ts` + 런타임 검증 스키마
- Firestore 문서 구조: 서버 전용 타입 + 저장/조회 함수
- 프롬프트 내용: `src/lib/prompts/*.ts`
- 편향 점수 계산: `src/lib/bias/scoring.ts`

> 원칙: 문서와 코드가 충돌할 경우, 배포 중인 코드 기준으로 먼저 맞추고 문서를 즉시 갱신합니다.

### 11-2. 스키마 버전 관리
- Step 1/2 결과, `anonymousSessions`, `diagnoses`, `analyses` 문서에는 `schemaVersion` 필드 추가 권장
- 응답 구조가 바뀌면 기존 문서를 직접 덮어쓰기보다 마이그레이션 함수로 변환
- 프론트가 참조하는 응답 타입은 optional 필드 남발보다 명시적 버전 분기 우선

**권장 예시:**
```typescript
type AnonymousSessionV1 = {
  schemaVersion: 1;
  status: "in_progress" | "completed";
  biasScores: number[];
};
```

### 11-3. 프롬프트 변경 규칙

- `step1-chat`, `step1-result`, `step2-analyze` 수정 시 프롬프트 버전 또는 changelog 기록
- 아래 항목 중 하나라도 바뀌면 회귀 확인 후 배포
- JSON 출력 형식
- 질문 톤/길이
- bias_map 생성 규칙
- Step 2 렌즈 선택 규칙

**최소 평가 항목**
- JSON 파싱 성공률
- 평균 응답 시간
- 평균 토큰 비용
- 동일 입력 대비 결과 일관성

### 11-4. 테스트 우선순위
- UI 스냅샷 테스트보다 아래 테스트를 우선 유지

- Step 1 시작 → 대화 진행 → 결과 생성 smoke E2E

### 11-5. 운영 로그와 관측성

### 11-7. 데이터 정리 정책
- `anonymousSessions`는 TTL 또는 스케줄러로 주기적 삭제

- 비용 절감을 위해 결과 재생성이 불필요한 경우 `resultSnapshot` 캐시 우선 사용
