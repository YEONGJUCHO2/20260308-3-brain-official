# 뇌피셜 QA 테스트 시나리오

## 목적

이 문서는 v1 출시 전 최종 점검용 체크리스트다.  
특히 아래 4가지를 확인하는 데 집중한다.

- 핵심 사용자 플로우가 끝까지 정상 동작하는지
- 비로그인/로그인 사용 횟수 제한이 서버에서 실제로 걸리는지
- Gemini 호출이 불필요하게 중복되지 않는지
- Firestore에 일일 사용량과 호출 로그가 정상 기록되는지

## 사전 준비

### 로컬 실행

cd C:\projects\3-brain-official
npm run dev

## 1. 비로그인 기본 플로우

### 목표

- 비로그인 상태로 Step 1 → Step 2까지 자연스럽게 진행되는지 확인
- 결과/리포트/공유 화면이 열리는지 확인

### 절차

1. 로그아웃 상태 확인
2. 랜딩에서 Step 1 시작
3. Step 1 질문을 끝까지 진행
4. Step 1 결과 화면 진입
5. 전체 리포트 진입
6. Step 2 입력 후 결과 확인
7. 대시보드 진입
8. 공유 카드 진입

### 기대 결과

- Step 1 결과가 정상 생성됨
- Step 1 전체 리포트가 정상 표시됨
- Step 2 결과가 정상 생성됨
- 대시보드에서 기존 결과를 다시 읽어옴
- 결과 화면/대시보드/공유 카드 진입 시 같은 결과를 다시 생성하지 않음

## 2. 비로그인 사용 횟수 제한

### 목표

- 비로그인 사용자가 하루 제한을 넘기면 서버에서 차단되는지 확인

### Step 1

1. 비로그인 상태에서 Step 1 완료
2. 다시 랜딩으로 돌아가 새 Step 1 시작

기대 결과:

- 두 번째 시작 시 차단 메시지 반환
- UI에서 에러가 깨지지 않고 보여야 함

### Step 2

1. 비로그인 상태에서 Step 2 결과까지 완료
2. 같은 날 다른 고민으로 Step 2 다시 실행

기대 결과:

- 두 번째 Step 2 분석은 차단되어야 함

### Firestore 확인

- `dailyUsage` 문서에 익명 scope 기준으로
  - `step1_starts = 1`
  - `step2_runs = 1`
  가 기록되는지 확인

## 3. 로그인 사용 횟수 제한

### 목표

- 로그인 사용자는 Step 2 하루 3회까지 가능한지 확인

### 절차

1. Google 로그인
2. Step 1 한 번 완료
3. 서로 다른 고민으로 Step 2를 3번 실행
4. 네 번째 Step 2 시도

### 기대 결과

- 1~3회는 성공
- 4회는 차단

### Firestore 확인

- `dailyUsage` 문서에 `uid`가 기록되는지 확인
- `step2_runs = 3`까지 증가하는지 확인

## 4. Step 1 하이브리드 질문 생성 점검

### 목표

- Step 1이 모든 턴마다 Gemini를 부르지 않는지 확인
- 하지만 질문 퀄리티가 너무 급격히 떨어지지 않는지 확인

### 절차

1. 새 Step 1 시작
2. 7턴 이상 답변
3. 서버 로그와 `dailyGeminiMetrics` 확인

### 기대 결과

- 기본 설정이면 Step 1 질문 Gemini 호출은 핵심 턴 위주만 발생
- 전체 Step 1 호출 수가 예전보다 줄어야 함
- 질문이 지나치게 반복되거나 부자연스럽지 않아야 함

### 추가 점검

- 환경변수 `STEP1_AI_TURNS`를 변경했을 때 실제 호출 수가 바뀌는지 확인

## 5. 중복 호출 방지 점검

### 목표

- 결과 화면/대시보드/공유 화면에서 같은 세션 결과를 다시 생성하지 않는지 확인

### Step 1

1. Step 1 결과 생성
2. 같은 세션으로 결과 화면 새로고침
3. 전체 리포트 진입
4. 공유 카드 진입

기대 결과:

- `dailyGeminiMetrics`가 불필요하게 다시 증가하지 않아야 함

### Step 2

1. Step 2 결과 생성
2. 결과 화면 새로고침
3. 대시보드 진입
4. Step 2 공유 카드 진입

기대 결과:

- 같은 고민 텍스트 기준으로 결과 재생성이 없어야 함

## 6. Gemini 실패 fallback 점검

### 목표

- Gemini 오류가 나도 서비스가 완전히 깨지지 않는지 확인

### 절차

1. 테스트용으로 `GEMINI_API_KEY`를 비우거나 잘못된 값으로 설정
2. Step 1/Step 2 진행

### 기대 결과

- Step 1은 질문 은행 fallback으로 진행 가능
- Step 2는 구조화 fallback 결과가 생성됨
- 앱이 500으로 죽지 않음

## 7. 호출 로그 점검

### 목표

- Gemini 호출이 일별로 기록되는지 확인

### Firestore 확인 포인트

- `dailyGeminiMetrics/{YYYY-MM-DD}`
  - `total_calls`
  - `models.*`
  - `sources.step1_question`
  - `sources.step1_result`
  - `sources.step2_analyze`

### 기대 결과

- 실제 사용 후 각 source 카운트가 증가해야 함
- 호출 수가 예상보다 급증하면 중복 호출 버그를 의심해야 함

## 8. 로그인 토큰 전달 점검

### 목표

- 로그인 사용자의 API 요청이 서버에서 로그인 사용자로 인식되는지 확인

### 절차

1. 로그인
2. Step 2를 여러 번 실행
3. `dailyUsage` 문서 확인

### 기대 결과

- 익명 scope가 아니라 `uid`가 있는 사용자 문서로 기록돼야 함

## 9. UI/UX 마감 점검

### Step 1

- 질문 카드가 모바일에서 잘리지 않는지
- 로딩 중 `AI 분석 중` 카드가 자연스러운지
- 결과 카드 상세 모달이 겹치지 않는지
- 차트 범례/라벨이 읽히는지

### Step 2

- 고민 입력 후 로딩 카드가 자연스러운지
- `생각 경고등` 안 편향 단어 강조가 읽기 쉬운지
- 렌즈 카드들이 너무 generic 하지 않은지

### 공통

- 깨진 한글이 남아 있지 않은지
- 공유 버튼/로그인 버튼이 dead button이 아닌지
- 광고 슬롯이 CTA처럼 보이지 않는지
- 광고 라벨이 분명히 보이는지
- 광고가 콘텐츠보다 먼저 눈에 띄지 않는지

## 10. 애드센스 점검

### 목표

- 애드센스 슬롯이 승인/운영 기준에 맞는 위치에만 있는지 확인
- 버튼 오인, 우발 클릭 유도, 과도한 광고 배치를 피하는지 확인

### 확인 위치

- Step 1 결과
- Step 2 결과
- 대시보드

### 기대 결과

- 랜딩 첫 화면에는 광고가 없다
- Step 1 결과에서는 광고가 `전체 리포트`와 `공유` 사이에만 있다
- Step 2 결과에서는 광고가 `선택된 분석` 본문과 `다음 행동 CTA` 사이에만 있다
- 대시보드에서는 광고가 `최근 성찰 기록` 아래에만 있다
- 광고 카드에 `Sponsored` 라벨이 보인다
- 광고가 로그인, 공유, 분석 시작 버튼과 시각적으로 경쟁하지 않는다

## 11. 최종 출시 판단 기준

아래를 만족하면 v1 공개 가능으로 본다.

- 비로그인 제한이 실제로 걸린다
- 로그인 제한이 실제로 걸린다
- Step 1/Step 2 결과가 정상 생성된다
- 결과 화면/대시보드/공유 카드에서 중복 생성이 없다
- `dailyUsage`, `dailyGeminiMetrics` 기록이 정상이다
- 치명적 깨짐, 무한 로딩, 500 에러가 없다

## 권장 테스트 순서

1. 비로그인 Step 1/2 완주
2. 비로그인 제한 초과 확인
3. 로그인 후 Step 2 3회/4회 확인
4. 애드센스 슬롯 위치/라벨 확인
4. 대시보드/공유 카드 중복 호출 확인
5. Gemini 키 제거 fallback 확인
6. Firestore 로그 확인
